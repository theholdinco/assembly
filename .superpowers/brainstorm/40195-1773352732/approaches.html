Three Approaches: Chat-Driven Deliverable Updates

How should the conversation feed into deliverable rewrites?

A

Conversation-Aware Rewrite

When user clicks "Update Deliverable", send the full chat history + current deliverable to a new prompt that rewrites the deliverable incorporating everything discussed. The prompt sees the whole conversation and decides what to change.

Pros

  • Most natural — AI sees the full context
  • Handles complex multi-turn corrections well
  • Single API call to produce updated deliverable

Cons

  • Larger prompt = higher token cost per update
  • Risk of unintended changes to unrelated sections
B

Extract-Then-Apply (Two-Step)

First, extract a structured list of changes from the conversation (like the current insight extraction). Then, apply those changes to the deliverable in a second pass — similar to the existing "Evolve" flow but sourcing from chat instead of insights.

Pros

  • User could review extracted changes before applying
  • More predictable — explicit list of what will change
  • Reuses existing evolution infrastructure

Cons

  • Two API calls = slower
  • Extraction step may miss nuance from conversation
  • Extra UI step interrupts the flow
C

Streaming Inline Update

When user clicks "Update", stream the updated deliverable directly into the page — replacing the current content in real-time. The deliverable page itself becomes the output surface, not the chat.

Pros

  • Most satisfying UX — user watches the update happen
  • No context switching between chat and deliverable

Cons

  • More complex frontend — need to handle streaming into the deliverable view
  • Harder to show "what changed" during streaming
  • If it goes wrong, the page flickers with bad content

Recommendation: Option A — simplest, most natural, and the "What Changed" section gives transparency without the overhead of a two-step process. Streaming (C) is a nice-to-have that can be added later on top of A.
